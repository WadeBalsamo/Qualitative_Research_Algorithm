SPECS ON LATEST REFACTOR: # QRA Modular Refactor — Three-Phase Plan

This refactor is split into three phases that ship independently.

- **Phase 1 — Freeze.** Freeze raw segmentation and validation testsets to disk, migrate legacy projects forward. CLI shape and orchestrator stage layout unchanged. (Already planned in detail below.)
- **Phase 2 — Testset Kinds and Content-Validity Refresh.** Add PURER (therapist cue blocks) and codebook (multi-label per participant segment) testsets on the frozen-testset foundation from Phase 1. Add a refresh subcommand for the existing VAAMR content-validity testset and add a new PURER content-validity testset. Codebook content-validity is deferred (codebook codes today have empty `exemplar_utterances`). Introduce two scoped CLI surfaces: `qra testset {create|refresh|list}` and `qra cv {create|refresh|list}`.
- **Phase 3 — Modular CLI Stages and Classification Overlays.** Decompose `process/orchestrator.py:run_full_pipeline` into independently invokable stages, expose them as `qra ingest`, `qra classify`, `qra assemble`, and separate classification results into overlay files at `02_meta/classifications/` so re-classification never touches the frozen `01_transcripts/segmented/` tree. `qra run` becomes a thin orchestrator over the stages.

After Phase 1, every existing project still runs `python qra.py run …` exactly as before; the only externally visible change is that re-runs reuse frozen segments and the AI answer keys for testsets refresh instead of the entire testset being regenerated. After Phase 2, two new testset kinds and PURER content-validity are available, and CV refresh is a single command. After Phase 3, every stage of the pipeline is independently runnable and re-classifying never re-segments.

Phase 1 introduces a separate, deletable legacy-migration module (`process/legacy_migration.py`) carrying every line of code that exists only to upgrade pre-modular projects. Cleanup of that module — and of the small back-compat seams introduced in Phase 1 — is **out of scope for this plan** and will be planned separately once in-flight projects have all been migrated.

# Phase 1 — Freeze Segmentation, Freeze Testsets, Legacy Import

## Context

`process/orchestrator.py:run_full_pipeline()` re-segments every transcript on every run, and `process/assembly/human_forms.py:export_validation_testsets()` re-samples testset segments on every run (re-shuffling the cohort pool every time, so the seed alone does not protect previously human-coded testsets — adding a single new transcript silently changes which segments appear). There is no on-disk artifact representing "the segmentation result" separately from `02_meta/training_data/master_segments.jsonl`, which mixes raw segmentation with classification output.

Phase 1 introduces two on-disk freeze layers:

1. **Per-session frozen segments** under `01_transcripts/segmented/<sid>/segments.jsonl`. Once a session is segmented, its segments are loaded from disk on subsequent runs instead of being recomputed. Adding a new transcript only segments that one transcript. Existing `participant_N` IDs stay stable via the already-cross-run-stable speaker map (`02_meta/speaker_anonymization_key.json`, loaded by `process/speaker_anonymization.py:load_speaker_map()`).
2. **Per-testset frozen artifacts** under `04_validation/testsets/<name>/`: a `manifest.json` (segment IDs + content SHA), a `segments_snapshot.jsonl` (text + identity at creation), a `human_worksheet.txt` (blind, never re-emitted), and a refreshable `AI_answer_key.txt`. The first time a testset is generated it freezes; on subsequent runs only the AI answer key is regenerated and the segment SHAs are verified against the current frozen segments.

Legacy projects that have a populated `master_segments.jsonl` and pre-modular testset worksheets are auto-migrated on first run by a clearly separated `process/legacy_migration.py` module.

## Non-goals for Phase 1

- No new CLI subcommands. `qra setup`, `qra run`, `qra analyze`, `qra testsets` keep their current shapes and behaviors. Phase 2 introduces `qra ingest`, `qra classify`, `qra assemble`, `qra testset {create|refresh|list}`, `qra cv {…}`.
- No PURER or codebook testset variants. Today only VAAMR participant testsets exist; that stays true through Phase 1. The freeze invariants are written to be kind-tagged (`kind="vaamr"`) so Phase 2 can extend them.
- No classification-overlay separation. Classification results still write to the in-memory `Segment` and out to `master_segments.jsonl` exactly as today.
- No setup-wizard changes.

## On-disk layout introduced

```
01_transcripts/
    diarized/                                 existing (unchanged)
    segmented/<session_id>/
        segments.jsonl                        NEW — frozen raw segmentation
        segmentation_meta.json                NEW — params_hash, segmenter version, ingest timestamp

04_validation/
    testsets/<name>/                          NEW shape — frozen
        manifest.json                         kind ("vaamr"), set_index, n_sets, seed, fraction,
                                               framework name+version, segment_ids, content_sha256
        segments_snapshot.jsonl               text + identity at creation

        AI_classification_testset_worksheet_1.txt      …
```

The names default to `vaamr_testset_<idx>` derived from the existing `set_idx` numbering in `process/assembly/human_forms.py:533`, so migrated projects get one folder per existing legacy worksheet pair.

# Phase 2 — Testset Kinds (PURER, Codebook) and Content-Validity Refresh

## Context (Phase 2)

Phase 1 froze the existing single VAAMR participant testset and made the validation-testset coordinator generic enough to accept a `kind` field in its manifest. Phase 2 lights up the two additional testset kinds the team has asked for and adds a refresh path for content-validity:

- **PURER testset (`kind=purer`)** — a held-out random sample of *therapist* segments that carry non-null PURER labels. PURER classification runs in `process/orchestrator.py` Stage 4b and propagates each cue block's primary/secondary onto every constituent therapist segment (lines 660-676), so the therapist segments in `master_segments.jsonl` already carry the labels needed.
- **Codebook testset (`kind=codebook`)** — a held-out random sample of *participant* segments that carry at least one entry in `codebook_labels_ensemble`.
- **Refreshable VAAMR content-validity** — today `process/assembly/human_forms.py:export_content_validity_testset` writes a content-validity testset only when it does not exist (line 595); there is no path to re-grade it against a new model without manually deleting and rerunning. Phase 2 adds `qra cv refresh` to regrade the AI answer key while keeping the human worksheet and definition key frozen.
- **PURER content-validity** — built from the PURER framework's `exemplar_utterances` / `subtle_utterances` / `adversarial_utterances`. The PURER framework definition (`frameworks/purer/definition.py`) already exposes those fields, and `classification_tools/llm_classifier.py:create_content_validity_test_set(framework)` already works against any framework that exposes them, so PURER plug-in is a one-line config change at the call site.
- **Codebook content-validity is deferred.** Codebook codes today have empty `exemplar_utterances` / `subtle_utterances` / `adversarial_utterances`. The code path is implemented and gated behind config; it stays disabled until the codebook content gap is filled. The deferral is a content gap, not an architecture gap — the same `qra cv create --framework codebook` surface added in Phase 2 will work without code changes once the codebook content is populated.

Phase 2 also generalizes the wizard and config so users can multi-select which testsets and which content-validity variants to generate, and exposes scoped CLI subcommands (`qra testset {create|refresh|list}` and `qra cv {create|refresh|list}`) so testset and content-validity operations can be performed without invoking the full pipeline. Stage decomposition and full-pipeline subcommands (`qra ingest` / `qra classify` / `qra assemble`) are explicitly **not** in Phase 2 — they are Phase 3.

## Goals

1. Generalize the Phase 1 frozen-testset shape to accept `kind ∈ {vaamr, purer, codebook}`. `manifest.json` carries a `kind` field; the AI answer-key writer and human-worksheet writer dispatch on it.
2. Move content-validity testset generation out of `process/assembly/human_forms.py` into a dedicated `process/assembly/content_validity.py` that uses the same Phase 1 freeze primitives. Add `refresh_cv_answer_key`.
3. Generalize wizard + config so users can multi-select which testsets and which CV variants to generate.
4. Add scoped CLI subcommands for testset and content-validity operations: `qra testset {create|refresh|list}` and `qra cv {create|refresh|list}`. Each loads segments from disk via Phase 1's `process/segments_io.py:read_master_segments` (no orchestrator decomposition required yet).
5. Keep `qra run` end-to-end: in Stage 7 it generates/refreshes every enabled testset variant and every enabled content-validity testset.

## Non-goals for Phase 2

- No orchestrator decomposition. `run_full_pipeline` stays monolithic; Phase 3 splits it.
- No `qra ingest` / `qra classify` / `qra assemble` subcommands; those are Phase 3.
- No classification-overlay separation. Phase 2 still writes classification fields onto in-memory `Segment` objects and through into `master_segments.jsonl` exactly as today.
- Codebook content-validity is not generated. The wizard offers it greyed-out; the `qra cv create --framework codebook` command exits with a deferral message.

# Phase 3 — Modular CLI Stages and Classification Overlays

## Context (Phase 3)

After Phases 1 and 2 the pipeline already has frozen segmentation, three frozen testset kinds, refreshable content-validity, and scoped `qra testset` / `qra cv` subcommands. What is still monolithic:

- **Re-running classification rewrites segment-level state.** Today every classifier in `process/orchestrator.py` (Theme stage at lines 411-606, PURER stage at lines 608-696, Codebook stage at lines 698-770) mutates `Segment` objects in place, and the orchestrator then writes the merged result to `02_meta/training_data/master_segments.{csv,jsonl}` (lines 845-919). There is no separation between "raw segmentation" and "classification annotations on top of segmentation". A user who wants to swap models for the theme classifier and rerun must invoke the entire pipeline.
- **`run_full_pipeline` is one ~1100-line function.** It cannot be re-entered at a stage boundary, every stage shares mutable local state, and observers can only watch the whole run.
- **There is no command surface for partial pipeline runs.** The only entry points are `qra run` (everything), `qra testset {…}` and `qra cv {…}` (testsets/CV only — Phase 2), and `qra analyze` (post-hoc Stage 8). There is no way to ingest only, classify only, or assemble only.

Phase 3 makes each pipeline stage a top-level callable with deterministic on-disk inputs and outputs, exposes those stages as `qra ingest`, `qra classify`, `qra assemble` subcommands, and separates classification results into per-classifier overlay files so re-classification is a strictly additive operation that never touches `01_transcripts/segmented/`. `qra run` becomes a thin wrapper over the stage subcommands.

## Goals

1. **Decompose `run_full_pipeline` into pure stage functions** under `process/orchestrator.py`. Each stage takes the pipeline config, optionally a list of `Segment` objects (or loads them from disk), and a progress observer; returns the modified segments (or `None` for side-effect-only stages). Each stage is independently invokable and idempotent on disk for the same inputs.
2. **Separate classification results into overlay files** at `02_meta/classifications/{theme,purer,codebook,cross_validation}_labels.jsonl`. Re-running a classifier overwrites only its overlay; the frozen `01_transcripts/segmented/` tree is never touched.
3. **`qra assemble` joins frozen segments + overlays** into `master_segments.{csv,jsonl}` and the per-session human-classification reports, and writes `02_meta/classifications/classification_manifest.json` recording which classifier versions and models produced each overlay.
4. **Add stage subcommands**: `qra ingest`, `qra classify [--what theme|purer|codebook|cross-validation|all]`, `qra assemble`. Each is a 5-15 line wrapper over a stage function.
5. **`qra run` becomes a thin wrapper** that calls the stages in order. Its CLI shape stays unchanged; existing config files keep working. `--resume-from` is preserved as an alias.
6. **Stage-level observability.** The existing `PipelineObserver` API (`on_stage_started/finished/progress`) gains an optional per-stage scope so partial runs report progress without referencing a non-existent prior stage.

## Non-goals for Phase 3

- No removal of legacy code paths or Phase 1 back-compat aliases. Cleanup is out of scope for this plan.
- No new testset kinds or CV variants beyond what Phase 2 delivers.
- No change to the segmentation algorithm, classifier prompts, ensemble logic, or any framework definition file.
- No change to the analysis package (`process/analysis/`); `qra analyze` keeps its current shape.

## Stage function decomposition

All stage functions live in `process/orchestrator.py` and follow this signature pattern:

```python
def stage_<name>(
    config: PipelineConfig,
    *,
    segments: Optional[List[Segment]] = None,    # passed when caller already has them
    framework: Optional[Framework] = None,       # required for theme/cv/assemble stages
    codebook: Optional[Codebook] = None,         # required when codebook is enabled
    observer: Optional[PipelineObserver] = None,
) -> Optional[List[Segment]]:
    """Each stage:
       1. Either receives `segments` from the caller, OR re-reads them from disk
          (frozen segments + overlays) if segments is None. This makes every stage
          standalone: `qra classify` doesn't need a prior in-memory ingest.
       2. Performs its work.
       3. Writes its on-disk artifacts (overlay file, master, reports, ...).
       4. Returns the (possibly mutated) segments list, or None for side-effect-only
          stages."""
```

```python
def run_full_pipeline(config, framework, codebook=None, observer=None) -> pd.DataFrame:
    segments = stage_ingest(config, observer=observer)
    if config.run_theme_classifier:
        segments = stage_classify_theme(config, framework, segments=segments, observer=observer)
    if config.run_purer_classifier:
        segments = stage_classify_purer(config, segments=segments, observer=observer)
    if config.run_codebook_classifier:
```

### Standalone re-entry semantics

Each stage's "no segments passed in" branch loads what it needs from disk:

```python
def stage_classify_theme(config, framework, *, segments=None, observer=None):
    if segments is None:
        segments = segments_io.load_segments_for_stage(
            config.output_dir,
            apply=("purer", "codebook", "cv"),    # everything EXCEPT theme
        )
    ...
```

The `apply` tuple lets each stage load only the overlays it needs and explicitly excludes the overlay it is about to overwrite (so prior theme labels don't bleed into a fresh theme classification run).

Per-stage `apply` defaults when called standalone:

| Stage | `apply` defaults |
|---|---|
| `stage_classify_theme` | `("purer", "codebook", "cv")` |
| `stage_classify_purer` | `("theme", "codebook", "cv")` |
| `stage_classify_codebook` | `("theme", "purer", "cv")` |
| `stage_cross_validation` | `("theme", "purer", "codebook")` |
| `stage_assemble` | `("theme", "purer", "codebook", "cv")` |
| `stage_validation_artifacts` | `("theme", "purer", "codebook", "cv")` |
| `stage_reports` | `("theme", "purer", "codebook", "cv")` |

This is what makes `qra classify --what theme` work without a prior `qra ingest`: the stage reads the frozen segments off disk, applies the non-theme overlays, runs the theme classifier, writes the theme overlay, and exits.

## Files to modify (Phase 3)

### `process/orchestrator.py` — full decomposition

The decomposition is the single largest mechanical change in this plan. The order of work below stages it so the orchestrator stays runnable end-to-end at every commit.

1. **Extract pure helpers first.** Pull the inline classifier-setup logic (lines 386-410, 608-625, 699-721) into module-private helpers (`_build_theme_classifier`, `_build_purer_classifier`, `_build_codebook_classifier`). No behavior change. These are used by both the stage functions and any future ad-hoc callers.

2. **Extract stage function bodies one at a time, top-down.** For each stage in the table above:
   - Move the stage's lines into a new `stage_*` function with the standardized signature.
   - At the bottom of the new stage function, call the appropriate `classifications_io.write_*_overlay` to persist results to its overlay file, then call `update_classification_manifest`.
   - Replace the original lines in `run_full_pipeline` with `segments = stage_*(config, ..., segments=segments, observer=observer)`.
   - Run end-to-end after each extraction; segment counts and report contents must match byte-for-byte (modulo the new overlay files).

3. **`stage_assemble` becomes the join point.** It takes the in-memory segments (which already carry classifier output from earlier stages in a `qra run`) and writes `master_segments.{csv,jsonl}`. When called standalone (`qra assemble`), it calls `load_segments_for_stage(run_dir, apply=("theme","purer","codebook","cv"))` to rebuild the in-memory state from frozen + overlays, then writes master. The master columns are unchanged — every existing column is still produced.

4. **`stage_validation_artifacts` writes the speaker key + per-session human-coding worksheets + flagged_for_review + Phase 2 testset/CV coordinators.** It accepts `segments` (already classified), so the existing logic at lines 921-1015 lifts unchanged.

5. **No segment dual-writes.** After this decomposition, classification fields are no longer written redundantly into both segment objects and overlays. The in-memory `Segment` continues to carry classifier fields (so downstream stages in the same `qra run` invocation see them without a disk round-trip), but the *persistent* record is the overlay; `master_segments.{csv,jsonl}` is recomputed each `stage_assemble` from frozen + overlays, not from a stale per-stage write.

### `qra.py` — new subcommands

```
qra ingest    -o DIR
                 [--reingest SESSION_ID]      Force re-segmentation of one session
                 [--reingest-all]              Force re-segmentation of every session

qra classify  -o DIR
                 [--what theme|purer|codebook|cross-validation|all]   default: all enabled in config
                 [--config FILE | --backend ... --model ...]
                 # If a per-classifier overlay already exists, it is overwritten
                 # (re-classification is intended to be the cheap operation).

qra assemble  -o DIR
                 # Reloads frozen segments + every overlay present, writes
                 # master_segments.{csv,jsonl} and 04_validation/human_classification_<sid>.txt,
                 # and refreshes existing testsets / CV (delegates to Phase 2 coordinators).
                 # Does NOT generate testsets that have not yet been frozen.

qra run       -o DIR ...                       Existing CLI; now a thin wrapper
                 [--resume-from STAGE_NAME]    Skip stages whose overlay/master is
                                                 already on disk; preserved as alias.
```

`qra ingest`:
- Validates that `data/input/` exists and has at least one `.vtt` or `_diarized.json`.
- Prints, per session, "frozen / reused" or "segmented / written".
- With `--reingest SID` or `--reingest-all`, bypasses Phase 1's freshness check by passing `force=True` to `segments_io.write_session_segments`.

`qra classify --what X`:
- If `--what` is omitted, runs every classifier enabled in the config.
- Refuses to run if `01_transcripts/segmented/` is empty (clear error: "no frozen segments; run `qra ingest` first").
- For each requested classifier, calls the corresponding `stage_classify_*` standalone (segments loaded via `load_segments_for_stage` with the per-stage `apply` defaults above).
- Writes/overwrites only that classifier's overlay and updates `classification_manifest.json`.
- Does NOT regenerate `master_segments.{csv,jsonl}` — `qra assemble` does that.

`qra assemble`:
- Refuses to run if no overlay files exist (clear error: "no classifications on disk; run `qra classify` first").
- Reads frozen segments + every present overlay.
- Writes master_segments, human-classification worksheets, flagged_for_review.
- Calls Phase 2's `generate_or_refresh_validation_testsets` and `generate_or_refresh_content_validity_testsets` with a `create_missing=False` flag (added to the Phase 2 coordinators in this phase) so it only refreshes existing testsets/CV; new testsets are still created via `qra testset create` / `qra cv create` (or via `qra run`).

`qra run --resume-from STAGE`:
- `STAGE` ∈ {ingest, theme, purer, codebook, cross_validation, assemble, validation, reports, analysis}.
- Skips stages strictly before `STAGE`. The skipped stages' on-disk artifacts must be present; otherwise `qra run` errors out and tells the user which subcommand to run first.

### `process/observer.py` — per-stage scope

The existing `PipelineObserver` ABC has `on_stage_started/finished/progress`. Add an optional `scope_id: Optional[str]` parameter so partial runs can scope their progress without referencing a non-existent prior stage. Default `None` preserves existing behavior. The default `LoggingObserver` and the wizard's TUI observer both ignore `scope_id` for back-compat.

### `process/config.py` — no new flags

`run_theme_classifier`, `run_purer_classifier`, `run_codebook_classifier`, `run_cross_validation`, `run_analysis` already exist. The stage subcommands consult these directly.

### `process/assembly/` — `create_missing` flag

`generate_or_refresh_validation_testsets` and `generate_or_refresh_content_validity_testsets` (added in Phase 2) gain a keyword argument `create_missing: bool = True`. When `False`, the coordinators skip kinds whose testset directory does not yet exist instead of creating them. `qra assemble` passes `create_missing=False`; `qra run` and `qra testset create`/`qra cv create` keep the default.

## Idempotency rules summary (Phase 3)

| Operation | Reads | Writes | Side effects |
|---|---|---|---|
| `qra ingest` | `data/input/` | `01_transcripts/segmented/<sid>/` (Phase 1 frozen) | Adds new sessions; idempotent on previously-frozen sessions unless `--reingest*` |
| `qra ingest --reingest SID` | `data/input/<sid>` | `01_transcripts/segmented/<sid>/` (force) | Overwrites only that session's frozen segments. Existing testsets that reference that session's segments will fail SHA check on next refresh — clear, intentional error surface. |
| `qra classify --what theme` | frozen segments, non-theme overlays | `02_meta/classifications/theme_labels.jsonl`, manifest | Idempotent on overwrite (overlay is fully rewritten each run) |
| `qra assemble` | frozen segments + every overlay | `master_segments.{csv,jsonl}`, `human_classification_<sid>.txt`, `flagged_for_review.txt`; refreshes existing testsets/CV | Idempotent; same inputs → same outputs |
| `qra testset create` (Phase 2) | master, segments | `04_validation/testsets/<name>/` (frozen) | Frozen — refuses to overwrite without `--force` |
| `qra testset refresh` (Phase 2) | master, manifest | `04_validation/testsets/<name>/AI_answer_key.txt` | Refreshes only the answer key; verifies sha first |
| `qra run` | everything | every artifact above | Composition of the above; behaves identically to the explicit chain |
